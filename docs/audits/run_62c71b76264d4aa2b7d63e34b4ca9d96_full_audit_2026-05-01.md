## Root Cause

The likely root was stale quant-control `RecoveryRequired` state from earlier same-instrument activity.

Evidence:
- The fresh MNG entry was allowed by authority because broker, journal, and ledger were flat.
- There was still robot-owned working-order residue from the prior transition.
- The quant-control phase could remain `RecoveryRequired` in that flat-with-owned-working-orders state.
- Because the phase remained stale, the fresh fill did not enter the normal bounded pending-alignment/protective-submit window.
- The protective audit then classified the first broker-visibility gap as `PROTECTIVE_MISSING_STOP` instead of `PROTECTIVE_PENDING_CONVERGENCE`.

This was not a broker-flat shutdown failure, but it was a bad handoff classification and recovery path trigger.

## Fix Applied

Source hardening applied after this audit:

- `QuantExecutionControlStore.TryClearRecoveryRequiredOnAuthoritativeFlat` now clears stale `RecoveryRequired` when broker, journal, and ledger are exposure-flat and all remaining working orders are robot-owned/adopted.
- The clear still refuses to run if there is broker exposure, journal exposure, ledger exposure, or external/untrusted working order evidence.
- The quant-control regression test now proves a fresh flat entry can move from stale `RecoveryRequired` to `PendingAlignment`.
- The protective audit regression now expects the immediate post-fill no-stop window to produce `PROTECTIVE_PENDING_CONVERGENCE`, not `PROTECTIVE_MISSING_STOP`.
- `RunSummaryBuilder` now falls back to daily robot summary error evidence if JSONL scan output misses a latest critical event.
- Core daily summary counting now treats `CRITICAL` as error-class, matching the NinjaTrader runtime behavior.
