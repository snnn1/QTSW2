# Run 002d688 MES Test Inject Audit

Audit date: 2026-05-01

Run:
- `runs/002d688079984e028f062e5aaa839847`
- Loaded DLL hash: `aba18016f948cf6cf0b3d00c918291064046767bc30c22075fec62f6dbbda1ca`

Question:
- MES appeared to buy and sell immediately after strategy startup.

Finding:
- Confirmed. This was not a normal timetable stream entry.
- The trade came from `Stream=TEST_INJECT`, `Instrument=MES`, intent `ce7a550c23048a54`.
- Entry was a market buy for quantity `1` at `2026-05-01T22:33:58.6445440Z`, fill price `6933`.
- Exit was target fill at `2026-05-01T22:34:01.0702561Z`, exit price `6935.25`.
- Time in trade was about `2` seconds.

Evidence:
- `logs/robot/robot_ENGINE.jsonl` emitted `TEST_INJECT_EXECUTED` at `2026-05-01T22:33:57.7610232Z`.
- `logs/robot/robot_MES.jsonl` emitted `ORDER_CREATED_MARKET_OR_LIMIT` with `order_action=Buy`, `order_type=Market`, `quantity=1`.
- `logs/robot/robot_MES.jsonl` emitted `EXECUTION_FILLED` with `side=BUY`, `order_type=ENTRY`, `position_effect=OPEN`, `stream=TEST_INJECT`.
- `state/execution_journals/2026-04-14_TEST_INJECT_ce7a550c23048a54.json` completed with `CompletionReason=TARGET`.

Root cause:
- `RobotSimStrategy.TestInjectTradeOnStart` was true for the MES strategy instance.
- The code default is false, so the likely cause is a saved NinjaTrader strategy template/workspace carrying the property as true.
- Before this fix, that saved UI setting alone was enough to submit a synthetic market entry in SIM/playback.

Safety result:
- The injected MES trade was protected, reached target, and closed flat.
- No evidence in this slice suggests an orphan position or rejected order from the MES injection.
- It is still unsafe for normal playback proof because synthetic `TEST_INJECT` activity contaminates runtime evidence.

Additional ownership finding:
- The same run later emitted `OWNERSHIP_CLASS_A_InvariantViolation` for `MES`, intent `ce7a550c23048a54`.
- Detail was `Negative remaining: entry=0 exit=1`.
- Root cause was ownership event-journal restore replay: separate `MappedEntryFill` and `MappedExitFill` rows for the same intent were replayed as overwrites instead of deltas.
- For a completed entry+exit intent, restart restore could create a slot with only the exit quantity, producing a false negative remaining invariant.
- This makes the run invalid as runtime proof even though the injected trade itself closed flat.

Fix applied after audit:
- `RobotSimStrategy` now requires a two-key guard:
  - NinjaTrader property `TestInjectTradeOnStart=true`
  - environment variable `QTSW2_ENABLE_TEST_INJECT` set to `1`, `true`, or `yes`
- If the strategy property is stale/true but the environment variable is absent, it logs `TEST_INJECT_SKIPPED` with reason `RUNTIME_OPT_IN_REQUIRED` and does not submit.
- Applied in both active strategy copies:
  - `system/RobotCore_For_NinjaTrader/Strategies/RobotSimStrategy.cs`
  - `system/modules/robot/ninjatrader/RobotSimStrategy.cs`
- `InstrumentOwnershipLedger.RestoreFromJournal` now accumulates repeated same-intent entry/exit restore rows instead of overwriting prior fill quantities.
- Added focused `OWNERSHIP_RESTORE` harness coverage for event-journal entry+exit rows, multiple exit rows, and aggregate fallback restore rows.
